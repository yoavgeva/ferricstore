# Pure ra Async WAL Benchmark

Measures raw `ra.pipeline_command` throughput with the async fdatasync WAL
patch, isolated from all FerricStore infrastructure (no Router, Batcher,
ETS, or Bitcask). Uses a trivial in-memory KV state machine.

## Analysis

The async WAL patch delivers a **132.2x speedup** in wait-per-write mode
over the blocking baseline. The key insight: while fdatasync blocks for ~20ms on macOS
(F_FULLFSYNC), the patched WAL continues accepting entries. When the sync completes,
all accumulated writers are notified at once. This amortizes the fdatasync cost across
47.3 writes per sync call instead of blocking the entire pipeline.

Fire-and-forget mode achieves 19,109,055 writes/sec (75232.5x),
showing the WAL's raw intake capacity when writers are not back-pressured.


## Methodology

- A **separate ra system** (`pure_ra_bench`) is started in an isolated temp
  directory, completely independent of the FerricStore application's ra system.
- The state machine is `Ferricstore.Test.KvMachine` -- a pure in-memory map.
  No disk I/O from the state machine; all measured I/O is from the ra WAL.
- The **only variable** is the WAL's fdatasync behavior: the patched WAL runs
  fdatasync asynchronously in a spawned process, allowing the WAL to keep
  accepting entries while the sync is in flight.
- The **known baseline** (254 writes/sec) was measured
  with the stock ra_log_wal that calls fdatasync synchronously in the batch
  processing loop (macOS, F_FULLFSYNC ~20ms per call).
