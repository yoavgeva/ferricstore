## Key Finding

**The hypothesis that larger batch windows improve throughput was disproven.**

Increasing `window_ms` from 1 to 100 produced only 15.8% variation in throughput,
with the baseline (1ms) performing best. This means the WAL fdatasync frequency is
**not the bottleneck** at ~266 writes/sec.

The actual bottleneck is likely:

1. **ra WAL GenServer serialization** -- all 4 shards share a single WAL process.
   Each `ra:pipeline_command` must pass through this single process, which serializes
   all writes regardless of batching.
2. **macOS F_FULLFSYNC** -- on Apple Silicon, fdatasync triggers F_FULLFSYNC which
   flushes the entire SSD write buffer (~1-3ms per call). This is a per-call fixed
   cost that batching cannot amortize when the WAL is already batching internally.
3. **Batcher GenServer.call backpressure** -- 50 writers calling `GenServer.call`
   on 4 Batcher processes creates contention. The p50 latency of ~174.1ms
   reflects queuing delay, not disk I/O time.

To improve write throughput beyond 266/sec, the bottleneck
to address is the single shared WAL process, not the batch window size.

Larger windows increase tail latency (p95/p99) because writes wait longer in the batcher before being flushed to Raft. The p50 may decrease if fsync amortization reduces per-write WAL overhead.

### WAL write strategy

The current benchmark was run with the **default** WAL write strategy.
The `wal_write_strategy` is set at ra system startup and cannot be changed at
runtime. To benchmark `o_sync`, add to `cluster.ex`:

```elixir
config = %{
  ...existing config...,
  wal_write_strategy: :o_sync
}
```

With `o_sync`, the WAL file is opened with `O_SYNC` flag, eliminating the
separate `fdatasync` syscall after each batch write. This can improve throughput
when the filesystem supports `O_SYNC` efficiently (most modern Linux kernels).

## Recommendations

Based on the benchmark results:

1. **Keep `window_ms=1` (default).** Larger windows did not improve throughput and
   only added latency. The default is already optimal.

2. **The WAL fdatasync is not the bottleneck.** At 266 writes/sec,
   the limiting factor is the single shared ra WAL GenServer, not fsync frequency.

3. **To improve write throughput beyond 266/sec, consider:**
   - Separate WAL per shard (ra system per shard instead of shared)
   - Bypass ra for single-node mode (direct Bitcask writes + fsync)
   - Use `:async` durability for latency-insensitive namespaces

4. **O_SYNC strategy:** May help on Linux where `O_SYNC` avoids a separate syscall.
   On macOS (F_FULLFSYNC), unlikely to make a significant difference. To test, add
   `wal_write_strategy: :o_sync` to the ra system config in `cluster.ex`.
