## Analysis

### Key Findings

1. **Caller-side overhead is 10us** (sandbox_key + shard_for + check_keydir_full +
   quorum_bypass_check + pipeline_command + write_version_incr). This is negligible.

2. **wait_ra_event accounts for 97.9% of total write time** (avg 567us).
   This is the time between `ra.pipeline_command` (non-blocking cast) and receiving
   the `{:ra_event, _, {:applied, _}}` confirmation. It includes:
   - WAL write + fdatasync (amortized across concurrent writers)
   - ra_server batch processing (log append, state machine apply)
   - StateMachine.apply/3 (Bitcask NIF append + ETS insert + PrefixIndex)

3. **Concurrent throughput comparison**: FerricStore achieves 8,165 writes/sec
   vs pure ra at 25,304 writes/sec (3.1x difference). The gap is
   entirely inside `apply/3` -- the StateMachine's Bitcask NIF write + ETS update.

4. **Per-write wait is 3.0x longer** with FerricStore than pure ra. Since both
   share the same WAL, the difference is the time the ra_server spends inside
   `StateMachine.apply/3` for each command in the batch.

### Bottleneck Identification

If ratio > 1, the bottleneck is in `Ferricstore.Raft.StateMachine.apply/3`:
- `NIF.v2_append_batch_nosync/2` -- appends to the Bitcask log file (disk I/O)
- `:ets.insert/2` -- updates the keydir ETS table
- `PrefixIndex.track/3` -- updates the prefix index ETS table

If ratio ~= 1, both systems are WAL-bound and the StateMachine cost is negligible.


## Methodology

- **Sequential profile**: Manually times each step using `:erlang.monotonic_time(:microsecond)`.
  Steps measured: sandbox_key resolution, shard_for routing, MemoryGuard check,
  NamespaceConfig durability lookup, ra.pipeline_command cast, receive ra_event,
  WriteVersion.increment.

- **Concurrent profile**: 50 Task.async writers each doing 100 writes via
  `ra.pipeline_command` + blocking receive. Measures only the full round-trip
  wait time per write.

- **Pure ra comparison**: Identical concurrent pattern but against an isolated
  ra system using `Ferricstore.Test.KvMachine` (in-memory map, no disk I/O
  from the state machine). The only difference is what happens inside the
  ra_machine `apply/3` callback.

- Both FerricStore and pure ra use the same WAL (patched or stock, depending
  on what the application loaded at boot). The comparison isolates the cost
  of the StateMachine (Bitcask NIF + ETS insert + PrefixIndex) from the cost
  of the Raft pipeline itself.
